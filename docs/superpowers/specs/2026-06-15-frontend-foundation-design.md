# modern-fmis Frontend Foundation — Design Spec

**Date:** 2026-06-15
**Status:** Approved for planning
**Scope:** Phase 2. Builds the React/TypeScript frontend walking skeleton — the **Client UI** wired to the existing backend — establishing the frontend architecture, runtime-config, data-layer, and testing patterns every later frontend feature copies. Backend foundation (Phase 1) is complete on `main`.

---

## Purpose

Deliver the smallest real frontend slice — managing **Clients** (list, create, detail) — through every frontend layer and tool, so the patterns are demonstrated concretely. Subsequent features (Farm, Field, activities) copy this template. The stack was fixed in the Phase-1 foundation spec; this phase decides and demonstrates the frontend *design*.

### Success criteria

- The frontend builds **once** into an immutable artifact and is configured at runtime via `config.json` (no per-environment rebuilds).
- The **Client** feature works against the backend contract: an authenticated user sees a list of clients, creates one via a dialog (landing on its detail), and views a client's detail.
- Every architectural seam, the runtime-config bootstrap, the auth guard (incl. deep-linking), the RTK Query data layer, and the centralized testing harness exist and are exercised by the Client slice.
- A developer adding the next feature can copy `features/clients/` without inventing new patterns.

### Explicit non-goals (deferred)

- **Playwright E2E and live integrated login.** A real end-to-end test drives a real browser through a real Auth0 login, and authenticated API calls need the backend's `[Authorize]` satisfied by a real token — both require a deployed **Auth0 tenant**, which arrives with the infrastructure phase. Stubbing auth in an "E2E" would just be a slow component test. So Playwright and genuinely-live login land in a later phase; this phase builds and **fully unit/component-tests the frontend in isolation** (test Auth0 seam + MSW).
- Farm/Field/activities UI, mapping/geospatial views, reporting. Later phases.
- Authorization (roles/permissions). The app authenticates only.

## Stack

| Concern | Choice | Notes |
|---|---|---|
| Framework | React + TypeScript (Vite) | SPA |
| Node | **24 LTS** | pinned via `.nvmrc` + `engines` |
| Package manager | **pnpm 11.7.0** | pinned via Corepack `packageManager` in `package.json` (the JS analog of `global.json`) |
| UI library | **MUI** (`@mui/material`); `@mui/x-data-grid` Community reserved for genuinely tabular, high-volume data later | small entities use a plain MUI `List`, not a grid |
| State / data | **Redux Toolkit + RTK Query** | RTK Query for server state (caching, mutations, invalidation); Redux slices available for client state when needed |
| Auth | **Auth0 React SDK** (`@auth0/auth0-react`) | authentication only |
| Contracts | hand-written **Zod** schemas (`z.infer` for types) — **no codegen** | drift caught by the Zod↔OpenAPI contract test |
| Unit/component tests | **Vitest + Testing Library + MSW** | jsdom; behavior-first |
| Test data | **`@faker-js/faker`** | via `ModelFactory` |
| E2E | **Playwright** | **deferred** (needs Auth0) |

## Build once, configure at runtime

The build is environment-agnostic and immutable. Runtime settings the build can't know (API base URL, Auth0 domain/clientId/audience) live in a **`config.json`** served next to the static assets, generated by the `application` Pulumi stack at deploy time.

- On startup, before rendering, the app **fetches `config.json` and validates it with a Zod schema** — malformed config fails fast with a clear error rather than a half-booted app.
- The validated config is provided via a small React context; RTK Query's base URL and the `Auth0Provider` settings are read from it.
- A committed `public/config.json` provides local-dev defaults; the deployed file overrides per environment. Same build deploys everywhere.

## App shell & bootstrap

`main.tsx` composes the app in this order (nothing renders until config is loaded and valid):

```
fetch config.json → Zod-validate → ConfigProvider
  → Auth0Provider        (domain/clientId/audience from config; onRedirectCallback restores deep-link returnTo)
    → Redux Provider     (store with the RTK Query api; base URL from config)
      → RouterProvider
```

## Routing & authentication

React Router (`react-router-dom`):

| Route | Purpose | Guarded |
|---|---|---|
| `/` | Forces authentication; redirects authenticated users to `/welcome` | yes |
| `/welcome` | Small landing page | yes |
| `/clients` | Client list + "New Client" create dialog | yes |
| `/clients/:id` | Client detail | yes |
| `/unauthorized` | Shown when authentication fails | **no (public)** |

**Auth guard (`<RequireAuth>`):**
- Authenticated → render the route.
- Unauthenticated → `loginWithRedirect({ appState: { returnTo: <current path + search> } })`.
- Auth error/failure → redirect to `/unauthorized`.

**Deep-linking:** the guard captures the originally-requested URL as `appState.returnTo`; `Auth0Provider.onRedirectCallback` navigates there after login. A link to `/clients/123` while unauthenticated → login → lands back on `/clients/123`. `/welcome` is only the default when there is no specific target (e.g. landing on `/`).

**Auth → API:** RTK Query's `baseQuery` reads the API base URL from config and attaches the Auth0 access token (`getAccessTokenSilently`) as a `Bearer` header — the frontend mirror of the backend `[Authorize]`.

## Data layer (RTK Query)

A feature owns its RTK Query **api slice** with endpoints for the Client operations, typed by the feature's Zod schemas:

- `getClients` → `GET /clients` → `ListResultModel<ClientResponseModel>`
- `getClient` → `GET /clients/:id` → `ClientResponseModel`
- `createClient` (mutation) → `POST /clients` (`CreateClientRequestModel`) → invalidates the client list so it refetches.

Loading/error states come from RTK Query hook flags; the create flow surfaces API `400` validation errors in the dialog (the backend is the source of truth; Zod also validates inline before submit).

## Feature-based structure

The frontend mirrors the backend's vertical slices. **Production code is feature-sliced; testing code is centralized** (see Testing).

```
src/
  app/         store, providers, router, runtime-config bootstrap
  shared/      cross-feature UI, the RTK Query base api, zod/config helpers
  features/
    clients/   components (list, detail, create dialog), the clients api slice,
               zod schemas, routes — everything for the Client feature in one folder
  testing/     ALL test support (see below) — not scattered per feature
```

## Client UI

- **List (`/clients`):** an MUI `List` — each item shows the client **name** (primary) and **email/phone** (secondary); clicking navigates to `/clients/:id`. Not a DataGrid (the entity is too small).
- **Create:** a **dialog** opened from "New Client". Fields: Name (required), Email, Phone, with the **email-or-phone-required** rule validated inline via Zod and API `400`s surfaced. On success → navigate to the new client's `/clients/:id`.
- **Detail (`/clients/:id`):** shows the client's fields; has room to grow (tabs/related data) in later phases.

## Error handling

- Zod validates form input client-side (incl. cross-field email-or-phone); the **API is the source of truth** — `400` responses are surfaced in the dialog.
- A missing client (`404`) renders a not-found state on the detail route.
- Auth failure → `/unauthorized`. Network/unknown errors → a clear inline error state (RTK Query error flags).

## Testing conventions

Behavior-first React testing — explicitly *not* the typical online approach (no shallow rendering, no mocking our own components, no asserting on state/props, no `getByTestId`-first).

**Principles**
- Assert on what a user perceives/does; query by **role/label/text** (`getByRole`/`getByLabelText`/`getByText`) — same vocabulary as the (future) Playwright locators.
- Render the **real component tree inside real providers** via a shared **`renderWithProviders()`** (fresh Redux store + RTK Query, router, config context, MUI theme, **test Auth0 provider**). Never `vi.mock()` our own modules.
- Stub **only the network edge** with **MSW** — the whole stack (component → RTK Query → fetch) runs for real; MSW is the real network handler for tests (the parallel to the backend's real InMemory DB at the persistence edge).
- `userEvent` (not `fireEvent`); `findBy*`/`waitFor` for async. **No snapshot tests.**

**Centralized harness — everything in `src/testing/`** (the frontend analog of `Fmis.TestSupport`; not scattered per feature, since stubbing is only at the API boundary):
- **`renderWithProviders()`** — wraps a component in the real providers using the shared `testConfig`.
- **`TestingApiServer`** — a config-aware wrapper over MSW:
  - `start()` / `reset()` / `stop()` lifecycle (msw `listen`/`resetHandlers`/`close`).
  - Per-endpoint setup methods (e.g. `setupGetClientList`, `setupGetClient`, `setupCreateClient`) that know the method+path and build the URL from the **shared `testConfig`** (the same config `renderWithProviders` feeds the app — one source of truth for the base URL).
  - Common `SetupEndpointOptions<TBody>` on every endpoint: `delayMs` (loading-state tests), `status` (error-state tests), `capture` (a `RequestCapture<TBody>`).
  - Per-endpoint response data (e.g. the list items, the created client).
- **`RequestCapture<TBody>`** — an object (not a callback) from `createRequestCapture<TBody>()` with `lastRequest` / `calls` / `wasCalled` / `callCount`; each setup types its `capture` to that endpoint's request body so `capture.lastRequest.body` is typed. Used to assert requests carry the expected data.
- **`ModelFactory`** — typed builders (`createClient(partial?)`, `createClientList(n)`, `createClientRequest(partial?)`) using **`@faker-js/faker`** for valid defaults merged with the caller's partial; types inferred from the Zod schemas. Tests assert only on fields they set (faker fills the rest, kept non-flaky).

**Contract test:** the **Zod↔OpenAPI** test asserts the frontend Zod schemas still match the backend's `/openapi/v1.json` (served unauthenticated, so no Auth0 needed) — verify against, never generate from.

## Local dev & packaging

- The frontend service joins **`docker-compose.yml`** (its own Dockerfile, serving the built static app). Note: fully-integrated login won't complete locally until a real Auth0 tenant exists (infra phase); component/contract tests cover behavior in isolation until then.

## Open questions (resolve during planning)

- Exact `config.json` schema fields and the local-dev default values.
- Whether the detail route fetches by id or reuses a list-cache entry (RTK Query selector) — likely fetch by id for deep-link correctness.
- MUI theme baseline (default vs a light custom theme) — default is fine for the foundation.
- Form handling approach within the dialog (React Hook Form + Zod resolver vs. controlled fields + manual Zod parse) — decide in planning.

## What later phases inherit

Farm → Field (mapping/geospatial UI begins) → crop seasons → field activities → reporting, each copying this feature-slice + RTK-Query + testing-harness template. Playwright E2E and live login land once the infrastructure/Auth0 phase is deployed.
