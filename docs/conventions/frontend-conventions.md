# Repository Rule: Frontend Conventions

**Applies to:** all frontend code (`frontend/`), every phase. Companion to [`architecture.md`](architecture.md); shared rules (TDD, no-codegen/Zod, method names contain a verb, no comments) apply here too.

## Toolchain

- **Node 24 (LTS)** — pinned via `.nvmrc` + `engines`. **pnpm 11.7.0** — pinned via Corepack (`packageManager` in `package.json`), the JS analog of `backend/global.json`. Run frontend commands from `frontend/`.
- React + TypeScript + **Vite**. UI: **MUI**. State/data: **Redux Toolkit + RTK Query**. Auth: **Auth0 React SDK**. Contracts: hand-written **Zod**. Tests: **Vitest + Testing Library + MSW**; data via **`@faker-js/faker`**.

## Build once, configure at runtime

The build is immutable and environment-agnostic. Runtime settings (API base URL, Auth0 domain/clientId/audience) come from a **`config.json`** served beside the static assets and generated by the `application` Pulumi stack per environment. The app **fetches and Zod-validates `config.json` before rendering** (fail fast on bad config) and provides it via context; RTK Query's base URL and `Auth0Provider` read from it. Never bake environment values into the build (no per-env rebuilds).

## App shell / bootstrap order

```
fetch config.json → Zod-validate → ConfigProvider → Auth0Provider → Redux Provider → RouterProvider
```
Nothing renders until config is loaded and valid.

## Routing & authentication

- React Router. Protected routes sit behind a `<RequireAuth>` guard; `/unauthorized` is the only public route (the auth-failure destination).
- Guard behavior: authenticated → render; unauthenticated → `loginWithRedirect({ appState: { returnTo: <current path+search> } })`; auth error → `/unauthorized`.
- **Deep-linking is required.** Capture the requested URL as `appState.returnTo` and restore it in `Auth0Provider.onRedirectCallback` — users land on the link they clicked, not a generic landing page. A bare landing (`/`) may default to a welcome page; specific links always return to themselves.
- RTK Query's `baseQuery` attaches the Auth0 access token as a `Bearer` header — the frontend mirror of the backend `[Authorize]`.

## Structure: feature-sliced production, centralized testing

Mirrors the backend's vertical slices.

```
src/
  app/        store, providers, router, runtime-config bootstrap
  shared/     cross-feature UI, the base RTK Query api, zod/config helpers
  features/<feature>/   components, the feature's RTK Query api slice, zod schemas, routes — together
  testing/    ALL test support (see below)
```

A feature owns its components, its RTK Query endpoints, and its Zod schemas in one folder. **Testing code is the one exception to feature-slicing — it is centralized** in `src/testing/`, because stubbing happens only at the API boundary and there's no value scattering it.

## Data layer

Each feature defines an RTK Query api slice typed by its Zod schemas: queries for reads, mutations for writes, with tag-based invalidation (a create invalidates the list). Loading/error come from the hook flags. Validate inputs with Zod client-side, but the **API is the source of truth** — surface its 4xx responses.

## UI

- **MUI.** Small entities use a plain MUI `List` (primary text + secondary sub-details), **not** `@mui/x-data-grid` — the grid is reserved for genuinely tabular, high-volume data.
- **Create via dialog**, then **navigate to the created item's detail** on success. Detail and list are routes (deep-linkable); create is a dialog over the list.

## Testing — behavior-first, real tree, stub only the edge

This is deliberately *not* the typical online approach. Rules:

- **Behavior, not implementation.** No shallow rendering, no asserting on state/props, no mocking our own components/modules. If a behavior-preserving refactor breaks a test, the test was wrong.
- **Query by role/label/text** (`getByRole`/`getByLabelText`/`getByText`); `getByTestId` is a last resort. Same vocabulary as Playwright locators.
- **Render the real tree in real providers** via the shared **`renderWithProviders()`** (fresh store + RTK Query, router, config context, MUI theme, test Auth0 provider).
- **Stub only the network edge with MSW.** The whole stack runs for real; MSW is the real network handler (the parallel to the backend's real InMemory DB). Auth0 is the one external seam (a test Auth0 provider supplies an authenticated user + stub token).
- `userEvent` (not `fireEvent`); `findBy*`/`waitFor` for async. **No snapshot tests.**

### Centralized harness (`src/testing/`)

- **`renderWithProviders()`** — real providers, using the shared `testConfig`.
- **`TestingApiServer`** — config-aware MSW wrapper: `start()`/`reset()`/`stop()`; per-endpoint setup methods (`setupGetClientList`, `setupCreateClient`, …) that build URLs from the **shared `testConfig`** (same config `renderWithProviders` feeds the app — one base-URL source of truth). Every endpoint takes `SetupEndpointOptions<TBody>` = `{ delayMs?, status?, capture? }` (loading, error, and request-capture testing) plus endpoint-specific response data. No test writes raw `server.use(...)`.
- **`RequestCapture<TBody>`** — an object from `createRequestCapture<TBody>()` exposing `lastRequest` / `calls` / `wasCalled` / `callCount`; each setup types `capture` to that endpoint's request body. Used to assert requests carry the expected payload. (It's an object you inspect, not a callback.)
- **`ModelFactory`** — typed builders (`createClient(partial?)`, `createClientList(n)`, …) using **faker** for valid defaults merged with the caller's partial; types inferred from the Zod schemas. A test asserts only on fields it set.

### Contract & E2E

- **Zod↔OpenAPI contract test** asserts the Zod schemas still match the backend's `/openapi/v1.json` — verify against, never generate from.
- **Playwright E2E** (real browser, real login) is added once a real Auth0 tenant exists (infrastructure phase) — it's not faked in earlier.
